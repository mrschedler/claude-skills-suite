# The multi-agent CLI coding playbook: Claude Code, Codex, and Gemini working in concert

**Developers are increasingly running Claude Code, OpenAI Codex CLI, and Gemini CLI as a coordinated team rather than picking one winner.** The emerging consensus — backed by benchmarks, practitioner reports, and a growing ecosystem of orchestration tools — is that each tool excels in a distinct role, and combining them produces measurably better outcomes than any single agent alone. A Milvus/Zilliz study found that pairing Claude and Gemini for adversarial code review caught **10 out of 15 bugs**, compared to Claude alone at 8 — and five models debating pushed detection from 53% to 80%. The practice is no longer theoretical: prominent developers like Simon Willison (Django co-creator) run "multiple terminal windows with different coding agents in different directories" as a daily workflow, and at least a dozen open-source orchestration tools now exist to manage these multi-agent setups.

## Proven orchestration patterns developers actually use

Five distinct patterns have emerged from practitioner communities, each serving a different workflow need.

**The "Architect + Workers" pattern** is the most widely adopted. Claude Code serves as the lead orchestrator — planning work, making architectural decisions, and handling complex multi-file refactoring — while Codex CLI and Gemini CLI handle specialized subtasks. Developer Sankalp describes this as "CC as the main driver, Codex for review and difficult tasks." The HalalLens guide formalizes it: "Use Claude Code as the 'architect' for planning and complex refactoring. Use Codex CLI for parallel test writing and rapid code generation. Use Gemini CLI for documentation, multimodal tasks, and broad codebase searches."

**The "Gemini-as-Context-Engine" pattern** exploits Gemini's **1M-token context window** as a read-only research tool piped into Claude Code. A developer on Composio documented updating their CLAUDE.md with instructions to call `gemini -p` (non-interactive mode) for codebase analysis. Tasks that previously failed after seven attempts with Claude alone completed in a single step when Gemini first loaded the full codebase context. Addy Osmani (Google Chrome engineer) endorsed this: "Ask Claude Code to use Gemini CLI to build a plan for Claude to take action on. This is useful with its 1M context window and free plan."

**The "Dual-Agent Detect/Fix Loop"** creates an autonomous code evolution system. Gemini CLI continuously scans the codebase and writes actionable items to a shared `todo.md` file — it never fixes, only detects. Claude Code monitors that file, claims tasks, implements fixes, and moves completed items to a done list. The filesystem acts as a simple message bus between agents.

**The "Adversarial Review" pattern** uses cross-model verification. In the Milvus study by Li Liu, five models independently reviewed the same pull request, then broadcast their reviews through five rounds of structured debate. The key finding: Claude + Gemini is the optimal two-model pairing for code review because "their weaknesses barely overlap." A developer on Hacker News reported using this for legal draft review: "The final version was shocking. The process found some flaws that surprised me."

**The "Slash Command Router" pattern**, documented by paddo.dev, builds `/gemini` and `/codex` commands within Claude Code that spawn the other CLIs and pipe results back. The routing logic: Claude handles backend logic and debugging, Gemini handles visual analysis and research with its large context window, and Codex handles architecture decisions and systems thinking. No MCP server required — just spawn the CLI and pipe back stdout.

## What each tool is objectively best at

Benchmark data and extensive user reports paint a clear picture of specialization. The tools are genuinely complementary rather than interchangeable.

**Claude Code dominates complex reasoning and multi-file refactoring.** It scores **80.8%** on SWE-bench Verified (Opus 4.6) and leads expert knowledge-work benchmarks by a wide margin (+289 Elo over Gemini on GDPval-AA). Practitioners consistently report it "gets vague requests right" — Nathan Lambert noted that "switching from Opus 4.6 to Codex 5.3 feels like I need to babysit the model." Claude's Agent Teams feature enables spawning parallel sub-agents with dedicated context windows, making it the natural orchestration hub. Its weakness is **token consumption**: Claude uses **3–4× more tokens** than Codex on equivalent tasks (6.2M vs 1.5M tokens on a Figma-to-code benchmark) and costs $5/$25 per million tokens for Opus.

**Codex CLI excels at speed, token efficiency, and terminal operations.** It scores **77.3%** on Terminal-Bench 2.0 for CLI/DevOps tasks and completes agentic tasks in roughly half the wall-clock time of Claude Opus. Its defining advantage is efficiency: GPT-5.2-Codex is described as "the most token-efficient model, consuming fewer tokens than any other model with similar capability." Despite slightly higher per-token pricing, Codex costs roughly **half** of Claude for equivalent tasks due to this efficiency. OpenAI's Codex received their first "High" cybersecurity classification, finding over 500 zero-day vulnerabilities in testing. Developer Nek12, with 400+ hours of experience across all three tools, reports: "Real thinking I only saw with GPT-5.1... Codex is really thinking when given a task." Its weakness: a smaller context window (256K–400K vs 1M), no sub-agent system, and it "won't ask clarifying questions even when instructed."

**Gemini CLI's value proposition is its 1M-token context window and free tier.** It offers **1,000 daily requests at zero cost** with the full context window — no other tool matches this. Developers use it primarily for codebase exploration, planning, and documentation rather than code generation. Gemini 3.1 Pro recently took the Terminal-Bench lead at **78.4%** and leads WebDev Arena for UI quality. Its context caching offers up to 75% cost reduction on repeated analysis of the same codebase. However, multiple practitioners issue serious warnings about using it for autonomous code writing. Developer Nek12 is blunt: "If you want Gemini to wipe your entire system or destroy your codebase — use Gemini CLI." The free-tier model (Gemini 2.5 Pro) scores only **63.8%** on SWE-bench Verified compared to Claude's 80.8%, a 17-point gap on complex tasks.

## Parallel execution and the tools that enable it

A rich ecosystem of orchestration tools has emerged in the past six months to manage multi-agent workflows. **Git worktrees are the universal isolation primitive** — every serious tool uses them to give each agent its own working directory and branch, preventing real-time file collisions.

**Claude Squad** is the most mature session manager, installable via `brew install claude-squad`. It manages multiple agents (Claude Code, Codex, Gemini, Aider, OpenCode) in separate tmux sessions with automatic git worktree creation. **Parallel Code** provides a GUI for running all three CLIs side-by-side with automatic branch isolation and mobile monitoring via QR code. **Zeroshot** offers the most automated orchestration — it routes tasks to different providers (`zeroshot run 123 --provider gemini --worktree --pr`) with support for GitHub, GitLab, Jira, and Azure DevOps issue backends.

For Claude Code-centric orchestration, **Gas Town** by Steve Yegge manages 20–30 parallel Claude Code agents with a "Kubernetes for AI agents" architecture including a dedicated Refinery agent for merge sequencing. **Oh-My-ClaudeCode** spawns real tmux worker panes with commands like `omc team 2:codex "review auth module"` and `omc team 2:gemini "redesign UI"`. **Claude Octopus** coordinates all three CLIs with a **75% consensus gate** — all three must agree before changes ship.

The simplest practical approach requires no special tooling. Developers run separate terminal tabs with each agent in its own git worktree, share context through plan files or CLAUDE.md/AGENTS.md, and merge sequentially. A developer at Composio demonstrated running headless Codex inside Claude Code via a slash command that invokes `codex exec` and pipes stdout back — Claude coordinates while Codex executes.

**Morph** offers a production-grade model router as a SaaS API — a task-complexity classifier trained on millions of coding prompts that routes to cheap vs. powerful models at $0.001 per request. **OpenRouter** provides a unified API for 500+ models with fallback routing and cost tracking, used extensively as a backend for multi-model setups.

## Cost savings from intelligent task routing are substantial

The financial case for multi-model workflows is compelling. A Particula Tech case study documented a **40-person team cutting a $12,000/month API bill by 40%** through tiered routing: Gemini Flash or Claude Haiku for simple edits (~$0.10–$0.50 per million output tokens), Gemini 3.1 Pro for standard coding tasks ($12/MTok), and Claude Opus for complex architecture only ($25/MTok).

The most accessible optimization is using Gemini's free tier for planning and exploration. Since Gemini offers 1,000 daily requests with the full 1M context window at zero cost, developers use it for specification writing, task decomposition, codebase exploration, and image-to-UI conversion — then hand implementation to Claude Code. Terry Cho documented this trend on Medium: "Because the price of Claude Code is high, there is a trend of using Gemini for planning and project management rather than main coding work." A developer tracked **10 billion tokens** over eight months on Claude's Max subscription plan, paying ~$800 versus an estimated $15,000+ on API pricing — a **93% savings** — but this still makes routing cheaper tasks to free/cheap models worthwhile.

Within Claude Code itself, using Sonnet 4.6 instead of Opus for daily work saves roughly **40%** on API costs while sacrificing only 1.2 percentage points on SWE-bench (79.6% vs 80.8%). For sub-agent tasks, specifying Haiku cuts costs further. Anthropic's Batch API offers 50% off for non-urgent workloads, and prompt caching can reduce costs up to **90%** on repeated context.

The token efficiency gap between tools is the hidden cost driver. Codex uses 3–4× fewer tokens than Claude on equivalent tasks. A Figma-to-code benchmark showed Claude consuming **6.2 million tokens** versus Codex's **1.5 million**. Despite Codex's slightly higher per-token price, its effective cost per task is roughly half of Claude's. The practical multi-model budget runs **$70–220/month** depending on intensity, compared to $100–200/month for Claude alone with heavier usage.

## Evidence that multi-model outperforms single-model is growing

The strongest evidence comes from the peer-reviewed **"Mixture-of-Agents Enhances Large Language Model Capabilities"** paper (Wang et al., ICLR 2025 Spotlight). It demonstrated that a layered architecture where models take outputs from other models as input scored **65.1%** on AlpacaEval 2.0 versus GPT-4o's 57.5% — using only open-source models. The key finding: models generate better responses when presented with outputs from other models, "even less capable ones." Cost-effective MoA variants delivered "performance comparable to GPT-4 Turbo while being 2× cheaper."

However, a counter-finding from "Rethinking Mixture-of-Agents" (also ICLR 2025) showed that Self-MoA — multiple outputs from the *same* top model — outperformed mixed-model MoA by **6.6%** on AlpacaEval. The key nuance: cross-model diversity helps **only when models perform similarly** on the task. When one model is clearly better, mixing in weaker models adds noise. This supports the task-routing approach rather than ensemble-everything: use the best model for each specific subtask rather than running all models on everything.

The Milvus adversarial debate study provides the most directly relevant evidence for coding. Five models reviewing independently and then debating lifted bug detection from **53% to 80%**. Claude + Gemini as a two-model pair caught 10/15 bugs because their error patterns are complementary. Critically, adding context *hurt* Claude's performance (53% → 47%) but dramatically helped Gemini (13% → 33%), suggesting the optimal pattern: let Claude reason independently while feeding Gemini additional context.

The SWE-rebench leaderboard adds an important insight: **"The harness matters more than the model."** The same model scores 23% with a basic scaffold and 45%+ with a sophisticated one — a **22+ point swing** that dwarfs the 1–2 point differences between frontier models. This means workflow design and orchestration quality are more impactful than model selection alone, directly supporting investment in multi-agent architecture.

## Failure modes and what experienced practitioners warn against

The most insidious failure mode is **"agentic drift"** — the gradual semantic divergence when parallel agents work on related code. Developer Helge Sverre describes it precisely: "It's not a merge conflict in the git sense — your files might merge cleanly. It's a semantic conflict. The code compiles, the tests pass, but you've built the same thing three times and each version encodes slightly different assumptions about how it should work." Tasks in a codebase are almost never truly independent; auth changes affect billing, API endpoints touch middleware.

**The merge tax is superlinear.** Dave Paola quantifies it: "5 parallel agents: frequent conflicts, some cascade. 9 parallel agents: a clusterfuck." N parallel branches create N(N-1)/2 potential conflict surfaces, and resolving each conflict changes the base for all remaining branches. His recommendation: run agents sequentially rather than in parallel for write-heavy work within the same project.

**Gemini CLI poses genuine safety risks when given write permissions.** Multiple independent reports confirm destructive behavior — deleted disks, wiped git history, replaced files with hallucinations. Nek12, with extensive experience across all three tools, warns: "Even if you deny it write permissions, it will find a way to destroy your files." The consensus is to use Gemini exclusively in read-only or non-interactive (`gemini -p`) mode.

**Context pollution** degrades all agents over time. Long sessions accumulate irrelevant exploration notes, test logs, and stack traces that reduce response quality. OpenAI's own Codex documentation warns: "If you flood the main conversation with noisy intermediate output, the session can become less reliable." Using `/clear` between tasks and `/compact` during long sessions mitigates this. An ICML 2025 paper found that keeping context files (CLAUDE.md/AGENTS.md) **under 300 words** is critical — verbose instructions actually degrade performance because agents spend more reasoning tokens processing instructions than solving problems.

**Configuration fragmentation** is a practical headache. Claude Code reads CLAUDE.md, Codex CLI reads AGENTS.md, and maintaining consistent instructions across both doubles the configuration overhead. Different agents also produce different coding styles — Claude tends toward clean, well-documented code with strong inline explanation, while Codex optimizes for execution efficiency and concise output, and Gemini often deviates from intended patterns. Over time, a codebase touched by all three can develop an inconsistent voice.

The overarching constraint is human review bandwidth. Simon Willison identifies the core bottleneck: "AI-generated code needs to be reviewed, which means the natural bottleneck on all of this is how fast I can review the results." Running eight agents when you can review two at a time creates expensive waste. Addy Osmani's framing captures the right mindset: "The sweet spot is a handful of background agents doing low-to-medium complexity work while you stay human-in-the-loop for architecture."

## Conclusion

The optimal multi-model strategy emerging from real practitioners converges on a clear hierarchy: **Gemini CLI (free) for exploration, planning, and context-heavy analysis; Claude Code (Sonnet) for daily implementation and worker sub-agents; Claude Code (Opus) for complex architecture, lead-agent orchestration, and multi-file refactoring; Codex CLI for code review, terminal automation, security checks, and token-efficient background tasks.** This tiered approach cuts costs 40–60% compared to running Claude Opus for everything, while producing measurably better outcomes through cross-model verification.

The most important insight is not which tool is "best" — they've converged within 1–2 points on major benchmarks — but that **workflow architecture matters more than model selection**. The 22-point performance swing from scaffold quality dwarfs model differences. Investing in orchestration patterns (git worktree isolation, adversarial review, task-appropriate routing) yields far greater returns than optimizing model choice alone. The ecosystem is only six months old and maturing rapidly, but the fundamental principle is already clear: treat AI coding agents like a team of specialists, not interchangeable generalists.
