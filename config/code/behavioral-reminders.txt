# Behavioral Protocol — Agent-Agnostic
# Single source of truth for how agents behave across all projects and frameworks.
# Injected by hook at SessionStart and PreCompact. CLAUDE.md handles tool-specific config only.

## PROBLEM-SOLVING

- root_cause_first=true | fix upstream, not symptoms | if patching multiple places, problem is higher
- first_principles=true | question current approach | best solution > current solution
- simplicity=true | simple+robust > complex+clever | excessive error handling = wrong design
- resolve_back=true | chase simplest fix through the chain, even if it means rethinking the current process

## BEFORE NON-TRIVIAL WORK

- clarify_first=true | ask questions 1 at a time, wait for answer | 5 questions prevent 50 wrong assumptions
- blast_radius=check | who/what does this affect? | butterfly wings
- confirm_understanding=true | state back, get user confirmation before executing

## WRITING FORMAT

| Consumer | Format | Example |
|----------|--------|---------|
| hook stdout | key=value | `grounding=false` |
| behavioral-reminders.txt | structured, terse | this file |
| CLAUDE.md | tables, bullets | no paragraphs |
| GROUNDING.md | decision tables, key facts | not narrative |
| ENGINEERING-NOTEBOOK.md | structured entries | date=, type=, tables for changes/decisions/why/impact, evidence |
| Obsidian notes | human prose OK | human is primary reader |
| Qdrant memories | self-contained narrative | semantic search needs context |

rule: agent-consumed files → scan-optimized. no filler, no transitions, no preamble.

## ENGINEERING NOTEBOOK FORMAT

### Entry format
```
## Entry N -- Title

date=YYYY-MM-DD
type=redesign|decision|finding|experiment|audit|fix

**Changes:**
| What | Before → After | Detail |

**Decisions:**
| Decision | Alternatives rejected | Reason |

**Why:**
| Driver | Before | After |

**Impact:**
| Target | Effect |

**Evidence:** Qdrant search hint, git ref, artifact DB record
```
- not all sections required. use what fits.
- date= and type= always required.
- evidence must include Qdrant search hint (not UUIDs).

### Index format
```
## Index
| # | Title | Date | Type | Line |
|---|-------|------|------|------|
```
- rebuilt on every entry addition
- positioned after header block, before first entry
- Line = actual line number in file

## SESSION LIFECYCLE

### On Session Start
1. Read GROUNDING.md at git root (if grounding=false in hook output, run /project-organize first)
2. gateway_call > rehydrate {topic, project_slug} — pulls relevant memories, context, recent activity
3. SESSION JANITOR (VITAL — you will want the user's task first; do this NOW even if it delays a minute)
   - Compare rehydrate results (+ hygiene block if present) against GROUNDING.md / PROGRESS.md / pipeline / git
   - Apply auto-heals within quota (see Session Janitor below). Supersede/update/confirm/tag only. Never delete.
   - Report: "Hygiene: {N fixed, M deferred, K need you}" then "Memory check: {N issues found}" if any remain
   - Prefer /rehydrate (janitor on by default) when orienting; skip only with --no-hygiene / --quick / --no-mcp
4. interagent_call > inbox {from: machine_id} — check for assignments from other machines
5. coordination_call > register_session {session_id, cwd, project, hostname}
6. Continue with user's task

### On PreCompact (primary persistence checkpoint)
Context is about to compress. Act NOW:
1. Persist unsaved decisions, findings, gotchas to appropriate stores (see Where Knowledge Lives)
2. Update coordination status if significant progress made
3. Check: did this session's work invalidate any memories retrieved earlier? Update before context is lost. This is inline — not a subagent.
4. Post progress to Mattermost thread if significant work completed
5. Summarize task state and next steps in output for post-compaction continuity

### On Session End
Hook outputs action items. Evaluate each:
- store_session_summary: memory_call > store if session produced meaningful work (>5 tool calls, real decisions)
- deregister_coordination: coordination_call > deregister_session
- post_mattermost_summary: reply to session thread with outcome
- github_sync: commit if uncommitted changes warrant it

## MEMORY PROTOCOL

### Think Before Storing
- Will a future session need this? If no, skip.
- Does it capture WHY not just WHAT? Include reasoning.
- Does an existing memory cover this? Search first. Update, don't duplicate.
- Consequence tier (also determines precedence when stores disagree):
  - High (patent, legal, decision-of-record, architecture): git files canonical. Qdrant is a pointer.
  - Medium (gotchas, preferences, ongoing decisions): Qdrant canonical. Git files point at it.
  - Low (session context, debugging): store if useful, let decay.
- Store conflict: investigate root cause first. Recency alone is not authority.

### Where Knowledge Lives -- reader-first
Each store answers ONE reader question. If two answer the same question, one is wrong.
Search stores before asking the user. For Qdrant, try 2-3 query variations.

| Reader's question | Store |
|------|------|
| Why does this project exist? What must I not do? | GROUNDING.md |
| How do I work in this specific repo? | CLAUDE.md (project) |
| How did we end up here? What was tried/failed? | ENGINEERING-NOTEBOOK.md |
| What's the technical architecture? | project-context.md (if present) |
| What's been discussed/decided (narrative, cross-project)? | Qdrant (memory_call) |
| What depends on what? Who/what is connected? | Neo4j (graph_call) -- merge_node/merge_relationship |
| Where's the full research report or spec? | MongoDB (mongodb_call), DB=homelab col=docs |
| What are the current sprints/tasks/questions? | Pipeline (project_call) |
| What findings/assessments has THIS project produced? | Artifact DB (artifacts/project.db) |

Preferences: Qdrant category=preference. pref_call legacy, reads only.

### Disambiguating examples -- tempting-but-wrong → correct
- Gotcha found mid-session → Qdrant, NOT CLAUDE.md (CLAUDE.md doesn't decay and isn't cross-project searchable)
- Architectural decision made today → Notebook entry (canonical record) AND Qdrant memory (searchable pointer). Not one or the other.
- Research report >1k words → MongoDB, NOT Qdrant (long text degrades embeddings; Qdrant stores the pointer)
- Project anti-pattern ("don't mock DB here") → GROUNDING.md §anti-patterns, NOT Qdrant feedback (durable project rule, not session context)

### Cross-Project Memory
Categorize Qdrant memories for the project that needs to FIND them, not the project you're working in.
Route all memory writes through memory_call > store (Qdrant). A PreToolUse hook (block-auto-memory.sh) blocks writes to ~/.claude/projects/*/memory/ at the tool layer.

**Auto-memory trap:** Claude Code's system prompt instructs agents to save to ~/.claude/projects/*/memory/ on user corrections/confirmations. Fires even with native memory disabled. Ignore it. Route all such saves via memory_call > store to Qdrant. Hook-enforced as of 2026-04-20 -- writes return a block decision; pivot to memory_call.

### Before Neo4j/Qdrant operations
Search "[tool] gotcha" in memory first. Known issues exist for both stores.

## SESSION JANITOR (every cold start — 2026-07-09)

Goal: every capable agent removes gunk from the *active surface* before user work. Not a full /memory-sleep pass.

### Quota
- Max 5 write actions (supersede / update / confirm / tag via update)
- Or stop after ~60s of hygiene work
- One-line report then continue the user's task
- If findings > ~10, spawn hygiene subagent with the same quota+tiers (do not unbounded-clean)

### Skip when
- Hygiene block severity=none and no semantic issues, OR
- Project already cleaned this session

### Auto-heal WITHOUT asking
| Action | Tooling |
|--------|---------|
| Broken supersede link / superseded_visible | memory_call > supersede |
| State memory contradicted by PROGRESS / pipeline / git (clear stale) | update or supersede → current truth |
| Import / low-density noise (openai-import, claude-import, empty shells) | update tags: add `gunk,exclude-from-default` (content unchanged if possible) |
| Still-accurate governing memory | memory_call > confirm |
| Additive cross-links / case-normalized graph node merges | graph_call as needed |

### Always ask / stop
- patent, protected, evolution-tagged subjects
- Two current sources disagree and both look intentional (true contradiction)
- memory_call > delete (never auto — prefer supersede/tag; bulk legacy → /memory-sleep)
- Large 0.85 cluster consolidate (defer to /memory-sleep or propose)

### Remove means
Out of default steering: supersede + tag. Not hard-delete. Aligns with /memory-sleep supersede-only.

### Boundary
- Session janitor: ≤5 actions, project-scoped, every cold start
- /memory-sleep: bulk/legacy/dream, dry-run default, Matt-approved execute
- Do not run a full sleep pass at session start

## HYGIENE AGENT

### What the spawned subagent does (when janitor findings > ~10 or inline would derail)
Scope: project memories only, not global. Same auto-heal tiers + quota as Session Janitor.
- memory_call > search by project tags/category
- memory_call > search filtered to last 7 days (recency pass)
- Compare claims against current state (GROUNDING.md, PROGRESS.md, pipeline, artifact DB, code)
- Auto-heal within quota; preserve supersede chain
- Tag never-retrieved import/low-density gunk; do not mass-delete
- Check engineering notebook for TOC block; flag if missing/stale
- Update Neo4j nodes if structural relationships changed
- Write authority: supersede/update/confirm/tag within project scope. Deletion and user-task findings out of scope.
- Report: {N fixed, M deferred, K need you}

## AS-YOU-GO MEMORY HYGIENE (added 2026-07-08 — session-start hygiene alone was not producing clean-as-you-go)

### Retrieval is a hygiene event
When a memory search returns a result you can SEE is stale, contradicted, or superseded-in-practice (doc/git/pipeline says otherwise), fix it IN THE SAME TURN: memory_call > update or supersede. Do not note it for later; later never comes. This is not optional polish — a stale memory you touched and left is a stale memory you endorsed. (Evidence 2026-07-08: one session doing this killed 9 stale memories that had survived months of sessions that "would clean up later," including a stale-but-popular memory with 22 retrievals steering agents wrong.)

### Retrieval feedback — evaluate the gathering, not just the gathered
When context-gathering from the MCP stack materially FAILS — semantic search misses something you later find by grep/other means, results are dominated by wrong-era memories, a needed fact existed nowhere — store ONE memory about it: tag `mcp-retrieval-feedback`, category `memory-system`, content = the query, what was missing, how it was eventually found. One per failure mode per session, not per query. These are aggregated weekly (memory-system Sprint 14) to fix search blind spots. Silence = the system never learns where it's blind.

## PROJECT SETUP

When grounding=false (no GROUNDING.md), run /project-organize before other work.
New projects need GROUNDING.md covering: (1) why it exists (2) key decisions and reasoning (3) anti-patterns (4) commercial model.

## LONG-ARC PROJECT STRUCTURE

Projects spanning weeks or months use these seven load-bearing artifacts. Other stores are secondary.

| Artifact | Role | Update cadence |
|----------|------|----------------|
| GROUNDING.md | WHY — goals, direction, where things are, constraints, anti-patterns, decisions. STATIC — an agent reads it to understand importance and direction, not current status | rarely |
| CLAUDE.md | HOW — agent conventions, tool paths for this repo | when conventions shift |
| PROGRESS.md | NOW — current state, sprint/phase status, recent changes, blockers. The ONLY root file that changes per session | per significant session |
| ENGINEERING-NOTEBOOK.md | JOURNEY — numbered dated entries, supersede don't delete | per significant session |
| artifacts/plans/current.md | NORTH STAR — active plan: phases, milestones, exit gates | per phase/pivot |
| artifacts/project.db | DATA — structured findings, experiments, reviews, decisions | per significant artifact |
| GOTCHAS.md | TRAPS — tactical operational hazards | when bitten |

### Plan doc is the spine
- Current plan lives at `artifacts/plans/current.md` (real file or symlink to active plan).
- Plan pivots create a new file. Old plan moves to `artifacts/plans/archive/` with supersession header.
- Never edit the plan doc in place after a pivot. Preserve history.
- Notebook entries and DB records reference plan docs by filename.

### Reading order for fresh agent
1. GROUNDING.md (why/forbidden)
2. CLAUDE.md (repo conventions)
3. PROGRESS.md (current state, blockers)
4. artifacts/plans/current.md (active phase, exit gate)
5. Last 3 ENGINEERING-NOTEBOOK entries (recent journey)
6. GOTCHAS.md scan before risky work
7. memory_call rehydrate for targeted topic

### Secondary layers — narrowed roles
- Pipeline (project_call): project-level metadata + phase transitions only. Not per-task.
- Qdrant: cross-project narrative, keystone memory per major phase.
- Neo4j: entity relationships at architecture scale.
- MongoDB: reference docs >1k words.

### Pipeline task scope
- In-project work → local docs + Qdrant + artifact DB. No pipeline task.
- Cross-project/cross-machine work (inbox class) → inbox + pipeline task at receiving project.

### PROGRESS.md role discipline (amended 2026-07-07; reverses the 2026-04-20 "obsolete" ruling)
PROGRESS.md is the canonical home for STATUS UPDATES. Session-to-session state changes go here — never into GROUNDING.md. GROUNDING.md stays static (goals, direction, decisions); if you're editing GROUNDING.md more than once a month, you're putting status in the wrong file.
Anti-drift rules (this is why it was briefly killed — don't repeat the failure):
- PROGRESS.md states WHERE we are; it does not duplicate the plan doc's phases/exit-gates (link to them) or retell notebook entries.
- Every update stamps a date. Stale sections get corrected or deleted, not appended around.
- CURRENT-STATE.md / STATUS.md / PROJECT-STATUS.md are non-standard aliases — merge into PROGRESS.md when touched.

## AFTER SIGNIFICANT WORK

If session produced decisions, architecture changes, gotchas, or blockers — persist them.
Think Before Storing criteria apply. Store what future sessions need, capture WHY, check for duplicates.
Dual-write: artifact DB (detailed, per-project) AND Qdrant (narrative, cross-project).
